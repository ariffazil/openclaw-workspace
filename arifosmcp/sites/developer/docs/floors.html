Constitutional Floors

The 13 constitutional floors form the governance backbone of arifOS. Every response must pass through each floor, with scores above the threshold required for a SEAL verdict.

F1–F13 Overview

Each floor evaluates a specific dimension of constitutional compliance. Floors with VOID triggers can reject a response entirely if violated. Other floors contribute to the aggregate score but do not independently cause VOID.

F1

Tauhid / Sovereignty

التوحيد — Divine Unity & System Sovereignty

The supreme floor. Ensures no response contradicts the foundational sovereignty principle. All governance authority derives from this floor.

θ = 0.99
VOID TRIGGER

NO VOID

Verdict States

After traversing all 13 floors, the governance kernel assigns one of these verdict states:

VerdictCodeMeaningAction
SEAL SEAL All 13 floors passed. Response is fully verified and constitutionally compliant. Response is sealed into VAULT999 Merkle chain and delivered to client.
PROVISIONAL PROVISIONAL All critical floors passed but some non-critical floors are borderline. Response delivered with advisory notice. Sealed with PROVISIONAL flag.
PARTIAL PARTIAL Most floors passed but one or more non-VOID floors failed. Response delivered with degraded confidence warning.
SABAR SABAR Patience verdict — insufficient evidence to determine. More time or data needed. Response held pending additional evidence gathering. Retry recommended.
HOLD HOLD Mechanical failure or insufficient conditions (NOT a constitutional violation). Response held. System retries or requests human intervention.
HOLD_888 HOLD_888 Judgment-level hold at stage 888_JUDGE. The APEX dashboard flagged an anomaly. Response held for APEX review. Requires manual inspection.
VOID VOID Constitutional violation detected. A VOID-trigger floor (F1, F3, or F13) failed. Response is REJECTED. Cannot be delivered. Logged with VOID reason.
Verdict Invariant — Critical Rule
VOID is ONLY for constitutional violations. Mechanical failures (timeouts, parsing errors, network issues, API failures) always produce HOLD, never VOID. This invariant is enforced at the kernel level and cannot be overridden by any tool, persona, or configuration. Violating this invariant would itself trigger a VOID on floor F1.
